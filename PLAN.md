# LinkChat 星座編碼方案計畫

## 文件目的

本文件只處理一件事：

- 如何在 `LinkChat` 儲存使用者的星座相關資料
- 如何把資料安全地傳到 `InternalAICopliot`
- 如何避免模型直接看到明顯的星座語彙，例如「太陽魔羯」

本文件刻意只聚焦在星座案例，不混入其他 trait system。

## 核心目標

這套方案要同時滿足以下目標：

1. `LinkChat` 可以保存使用者的星座相關結構化資料
2. `LinkChat` 傳給 `InternalAICopliot` 的資料不直接暴露星座語意
3. `InternalAICopliot` 擁有真正的理論解釋權與 prompt 組裝權
4. 最終送給 AI 的內容不直接包含「太陽」「月亮」「上升」「魔羯」等容易觸發模型既有知識的詞
5. 未來若私有理論調整，應盡量只修改 `InternalAICopliot`

## 產品定位補充

本專案中的 AI 功能，定位上更接近：

- 軍師
- 顧問
- 分析與建議工具

而不是：

- 即時聊天產品
- AI 代替使用者和他人對話的系統

這個定位不只適用於關係互動，也適用於更廣的人際判斷場景，例如：

- 判斷某人是否可能融入團隊
- 判斷某種互動方式是否適合某個合作情境
- 協助使用者整理對某人的觀察方向

目前不把產品定義成可直接聊天的系統，主要原因是：

- 即時聊天成本高
- 對話狀態管理複雜
- 長上下文維護成本高
- 生成錯誤的風險更直接

未來是否支援直接聊天可以再討論，但本文件目前以軍師型產品為前提。

## 問題定義

若直接把「太陽魔羯」這種字串送進模型，模型很可能會套用它原本學過的大眾星座知識。

但本專案的前提是：

- 使用的不是大眾版星座解讀
- 而是個人私有理論
- 而且這套理論與模型原本知道的內容可能差很大

因此這個案例不能採用：

- 直接把原始星座名詞送給 AI
- 直接把 `LinkChat` 當成 prompt 組裝中心
- 直接把私有理論 prompt 存在 `LinkChat`

## 最小資料模型

### `LinkChat` 應保存的資料

建議以使用者為中心，保存星座代碼資料。

範例：

```json
{
  "userId": "u_001",
  "theoryVersion": "astro_v1",
  "traits": {
    "sun": "a1",
    "moon": "b2",
    "rise": "c4"
  },
  "updatedAt": "2026-03-16T12:00:00Z"
}
```

這裡的設計重點：

- `sun`、`moon`、`rise` 是結構位置，不是敏感理論內容
- `a1`、`b2`、`c4` 是 opaque code
- `LinkChat` 不需要知道這些 code 最後如何寫成 prompt

### `InternalAICopliot` 應保存的資料

`InternalAICopliot` 是私有理論映射中心。

建議至少具備以下 mapping 能力：

```json
{
  "theoryVersion": "astro_v1",
  "slot": "sun",
  "code": "a1",
  "interpretation": "此人在核心自我表現上傾向先建立控制感，再逐步暴露真實想法。",
  "promptFragment": "請把這項特質視為一種偏慢熟、先防守後投入的核心人格動能，不要套用一般星座敘事。"
}
```

這裡的設計重點：

- `slot + code + theoryVersion` 才是完整查找鍵
- `interpretation` 是內部中性解釋
- `promptFragment` 是真正供 prompt 組裝使用的資產

## 為什麼 `slot` 可以直接用 `sun`、`moon`、`rise`

這些字不是最敏感的部分。

真正會讓模型偏掉的，是：

- 看到明確星座詞
- 再把那些詞套進它既有的大眾占星知識

而 `sun`、`moon`、`rise` 在這套設計中只是系統內部結構鍵，用途是：

- 區分哪一個欄位位置
- 幫助 `InternalAICopliot` 查對應 mapping

只要最終送進模型前，不把它們重新渲染成「太陽魔羯」這種語言，風險就比直接送原詞低很多。

## 傳輸契約

`LinkChat -> InternalAICopliot` 的最小 payload 應類似：

```json
{
  "userId": "u_001",
  "theoryVersion": "astro_v1",
  "traits": {
    "sun": "a1",
    "moon": "b2",
    "rise": "c4"
  }
}
```

```text
                    傳輸契約責任邊界
┌──────────────────────────────────────────────────┐
│                                                  │
│  ✔ 允許傳送                ✘ 禁止傳送             │
│  ────────────              ────────────           │
│  ├ userId                  ├ 解釋 code 代表什麼   │
│  ├ theoryVersion           ├ prompt wording       │
│  └ traits                  ├ 最後送給模型的表述    │
│    ├ slot (sun/moon/rise)  ├ 太陽魔羯             │
│    └ code (a1/b2/c4)       ├ 月亮雙魚             │
│                            ├ 上升天蠍             │

       │                            │
       ▼                            ▼
  告訴 InternalAICopliot        這些責任屬於
  目前有哪些代碼 +              InternalAICopliot
  該用哪個版本解碼              不屬於傳輸層
```

## `InternalAICopliot` 內部轉換流程

收到 payload 後，應依照以下流程：

```text
┌─────────────┐
│  收到 payload │
└──────┬──────┘
       ▼
① 讀取 theoryVersion
       │
       ▼
② 逐一處理 traits 中的 slot → code
       │
       ▼
③ 用 theoryVersion + slot + code 查 mapping
       │
       ▼
④ 取得中性解釋與 prompt fragment
       │
       ▼
⑤ 將多個 fragment 組合成統一的內部人物描述
       │
       ▼
⑥ 把這份內部描述送入 AI prompt
       │
       ▼
┌─────────────────────────────────┐
│ 模型看到的是你定義過的結果        │
│ 不是自己從星座詞出發進行推論      │
└─────────────────────────────────┘
```

## 軍師型輸出方向

目前共識是：

- 輸出偏向分析到建議之間
- 不直接把底層 prompt 或理論內容暴露給使用者
- 不直接提供可以原封不動拿去發送的聊天文案
- 不把系統做成代替使用者發話的工具

在回答形式上，先採用骨架式輸出思路，而不是過早寫死單一文風。

目前較合適的骨架方向為：

1. 觀察角度
2. 可驗證方向
3. 風險提醒

這樣做的原因是：

- 比較不容易變成罐頭回答
- 比較不容易變成空泛雞湯
- 比較不容易直接替使用者下結論
- 也比較不容易不小心把底層 prompt 暴露出來

實際輸出文風與呈現細節，仍需後續測試，不在本文件先寫死。

## 真正應該避免的錯誤做法

### 錯誤做法一：`LinkChat` 直接保存完整 prompt

這會導致：

- 產品層知道太多私有理論細節
- 兩個專案強耦合
- 之後改理論時需要同步改多處

### 錯誤做法二：`LinkChat` 直接傳「太陽魔羯」

這會導致：

- 模型容易套用既有知識
- 私有理論失去控制權
- prompt 偏差風險升高

### 錯誤做法三：用純數字 `tag=1`

這會導致：

- 語意不清楚
- 維護成本高
- 擴充困難

### 錯誤做法四：用隨機 UUID 當主要理論鍵

UUID 可以作為文件 id，但不適合作為主要理論識別碼，因為：

- 人工維護困難
- 看資料時不可讀
- 跨系統排查問題很痛苦

## 文件 id 與業務識別碼的切法

Firestore 的文件 id 可以不是 UUID。

若是字典型資料，文件 id 直接使用穩定字串是可行的，例如：

- `sun`
- `moon`
- `rise`

但要注意：

- `sun` 是 slot，不是 code
- `a1` 才是該 slot 下的內部分類代碼

因此不應混成：

- 用 `sun` 同時表示位置與分類

正確做法是保留兩層：

- slot
- code

## 為什麼這樣分工最穩定

```text
LinkChat                                      InternalAICopliot
   │                                                 │
   ├─ 保存使用者結構化資料                           ├─ 持有 code mapping
   ├─ 決定本次送哪些 slot / code                     ├─ 持有私有理論解釋
   └─ 不保存私有 prompt                              ├─ 持有 prompt fragment
                                                     └─ 組裝給 AI 的最終語意
   │                                                 │
   └───────────────────── 跨專案契約 ─────────────────┘
                 theoryVersion + slot + code
```

這樣的好處是：

- 理論修改集中在一個地方
- 跨專案契約穩定
- 模型偏差風險較低
- `LinkChat` 不會被 prompt 細節污染

## 正式實作範圍

本方案是一次到位的正式資料切法，不採分階段版本。

系統必須直接具備以下能力：

1. `LinkChat` 保存完整星座結構資料：
   - `theoryVersion`
   - `traits.sun`
   - `traits.moon`
   - `traits.rise`

2. `LinkChat` 對 `InternalAICopliot` 輸出固定契約：
   - `userId`
   - `theoryVersion`
   - `traits`

3. `InternalAICopliot` 必須以：
   - `theoryVersion + slot + code`
   作為標準解碼鍵

4. `InternalAICopliot` 必須持有：
   - 中性解釋
   - prompt fragment
   - 私有理論 mapping

5. AI 最終只看到：
   - 私有理論轉譯後的中性描述
   - 組裝完成的 prompt 語意

本方案明確不採用以下做法：

- 直接把原始星座詞送進模型
- 讓模型自行猜測分類
- 用數字 tag 當主要跨系統鍵
- 把完整 prompt 存在 `LinkChat`
- 把理論解釋分散到兩個專案

## 可延伸但不改變核心方案的項目

以下內容未來可以擴充，但不構成另一個階段版本，也不改變本文件定義的核心資料切法：

- 增加更多星座位置
- 增加 code 關聯規則
- 擴充 theory version 管理細節
- 將 prompt fragment 拆得更細
- 加入 consult log，追蹤本次分析用了哪些 slot/code

## 目前已知困難與待討論事項

本方案目前已知有以下難點，這些難點應直接被視為產品與 prompt 設計議題，而不是略過不談：

### 困難一：回答太保守時，使用者會覺得很罐頭

若輸出只剩下安全、空泛、誰都能套的建議，使用者會明顯感受到內容沒有用。

### 困難二：回答太具體時，容易變成在替使用者下結論

若系統過度具體地描述某人「就是怎樣」，風險包括：

- 分析過度武斷
- 使用者把系統當成權威判決
- 系統輸出失去可驗證性

### 困難三：回答太深入時，可能洩漏底層 prompt

若分析文字與底層 prompt 過度接近，可能導致：

- 私有理論外露
- prompt 被使用者反推出來
- 系統價值被直接複製

### 困難四：直接給文案雖然方便，但風險偏高

直接提供可發送文案雖然看起來很有用，但它容易：

- 讓產品滑向代聊工具
- 提高誤用風險
- 讓使用者忽略情境差異

因此目前傾向以分析與建議為主，不主打直接代寫聊天內容。

### 困難五：不同分析主題可能需要不同 prompt 結構

例如：

- 團隊融入
- 人際互動
- 合作判斷

這些主題未必能共用同一套 prompt 模板。

因此分析入口未來可能適合用：

- 單選
- 下拉
- 多選

來區分不同任務類型，再分別注入對應 prompt。

但這部分仍需測試，不在本文件先寫死最終 UI 與 request 形式。

## 本文件的最終結論

本案例的正確資料切法是：

- `LinkChat` 保存 `slot -> code`
- `LinkChat` 傳輸 `slot + code + theoryVersion`
- `InternalAICopliot` 保存 `slot + code + theoryVersion -> 私有解釋 / prompt fragment`
- AI 最終只看到經過私有理論轉譯的中性描述

這是目前最能兼顧：

- 架構清晰
- 理論保密
- 維護成本
- 模型偏差控制

的方案。
