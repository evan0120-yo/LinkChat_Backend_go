# LinkChat 星座編碼方案開發文件

## 文件目的

本文件定義 `LinkChat` 在星座案例下的實作規則。

本文件只處理：

- `LinkChat` 如何保存星座編碼資料
- `LinkChat` 與 `InternalAICopliot` 的傳輸契約
- 哪些資料可以進模型
- 哪些資料不可以進模型

本文件不討論其他 trait system，不討論泛化架構命名，只聚焦本次星座方案。

## 核心原則

### 原則零：產品定位是軍師，不是即時聊天工具

本專案中的 AI 功能，現階段定位為：

- 軍師
- 顧問
- 分析與建議系統

不是：

- 即時聊天工具
- AI 直接代替使用者和對方對話的系統

原因不是理念性排斥，而是工程與成本判斷：

- 即時聊天成本更高
- session 與上下文管理更複雜
- 錯誤回覆會更直接地作用在真實互動上

因此目前開發規則應以軍師型輸出為前提。

### 原則一：`LinkChat` 只保存結構化代碼，不保存私有 prompt

`LinkChat` 內允許存在的資料是：

- 使用者對應的星座代碼
- 使用中的理論版本
- 必要的欄位結構

`LinkChat` 內不應保存：

- 給 AI 的最終 prompt 文字
- 私有理論長文說明
- 會直接暴露理論內容的完整 mapping 描述

### 原則二：跨專案契約必須穩定且可讀

`LinkChat -> InternalAICopliot` 之間的資料契約，必須使用：

- `theoryVersion`
- `slot`
- `code`

不得使用：

- 純數字 `tag`
- 臨時生成的隨機 id 當主要業務識別碼

原因：

- 純數字語意太弱
- 隨機 id 難以維護
- 排查 mapping 問題時不易閱讀

### 原則三：真正的解釋權在 `InternalAICopliot`

`LinkChat` 不負責說明：

- `sun + a1` 的理論意義是什麼
- 該怎麼被寫成 AI prompt

`InternalAICopliot` 才負責：

- 查 mapping
- 取得中性人物解釋
- 組裝最終 prompt

### 原則四：模型不得直接看到原始星座語彙

若此功能的目的是避免模型套用大眾占星知識，則最終送入模型的內容必須避免直接包含：

- 太陽
- 月亮
- 上升
- 魔羯
- 雙魚
- 其他原始星座語詞

若有例外，必須是經過明確設計與驗證的特例，而不是預設做法。

## 建議資料結構

### `LinkChat` 使用者星座資料

建議保存格式：

```json
{
  "userId": "u_001",
  "theoryVersion": "astro_v1",
  "traits": {
    "sun": "a1",
    "moon": "b2",
    "rise": "c4"
  }
}
```

欄位規則如下：

- `userId`
  - 使用者識別碼

- `theoryVersion`
  - 此筆資料使用哪一版星座理論
  - 未來版本切換時不可省略

- `traits`
  - 星座相關結構化資料容器

- `traits.sun`
  - `sun` slot 對應的 code

- `traits.moon`
  - `moon` slot 對應的 code

- `traits.rise`
  - `rise` slot 對應的 code

### `InternalAICopliot` mapping 資料

建議保存格式：

```json
{
  "theoryVersion": "astro_v1",
  "slot": "sun",
  "code": "a1",
  "interpretation": "此人在核心自我表現上傾向先建立控制感，再逐步暴露真實想法。",
  "promptFragment": "請把此特質視為慢熟與保守的核心人格動能，不要套用一般星座語彙。"
}
```

查找鍵規則：

- `theoryVersion + slot + code`

不得只靠：

- `code`
- `slot`
- 文件自動 id

單獨做查找。

## 文件 id 規則

### slot 類型資料

對於 slot 字典或固定欄位類型資料，Firestore 文件 id 可以直接使用穩定字串，例如：

- `sun`
- `moon`
- `rise`

這是可接受的，因為它們是固定結構位置。

### code 類型資料

對於理論分類值，不建議使用：

- 純數字 tag，例如 `1`
- 隨機 UUID

建議使用穩定且可讀的代碼，例如：

- `a1`
- `b2`
- `c4`

或未來若需要更明確，可使用：

- `sun_a1`
- `moon_b2`

但無論採哪種格式，都要保證：

- 對人可辨識
- 對系統穩定
- 跨專案一致

## API 傳輸規則

`LinkChat` 對 `InternalAICopliot` 的 request body 應只包含解碼所需資料。

建議格式：

```json
{
  "userId": "u_001",
  "theoryVersion": "astro_v1",
  "traits": {
    "sun": "a1",
    "moon": "b2",
    "rise": "c4"
  }
}
```

### 傳輸層允許內容

- `userId`
- `theoryVersion`
- `traits`
- `slot -> code`

### 傳輸層禁止內容

- `太陽魔羯`
- `月亮雙魚`
- `上升天蠍`
- 私有 prompt 全文
- 大量理論說明文字

## Prompt 組裝規則

### `LinkChat` 端禁止事項

`LinkChat` 不得：

- 直接組最終 AI prompt
- 把 `sun + a1` 寫成完整人格敘述後再當正式知識輸出
- 把私有理論 mapping 硬寫在產品模組中

### `InternalAICopliot` 端責任

`InternalAICopliot` 的 Prompt 組裝管線如下：

```text
┌─────────────────────────────────────────────────────────┐
│           InternalAICopliot Prompt 組裝管線              │
│                                                         │
│  ① 收到 theoryVersion                                   │
│  │                                                      │
│  ▼                                                      │
│  ② 逐一讀取 traits（遍歷每個 slot → code）               │
│  │                                                      │
│  ▼                                                      │
│  ③ 用 slot + code + theoryVersion 查 mapping            │
│  │                                                      │
│  ▼                                                      │
│  ④ 轉成中性敘述（不含星座語彙）                           │
│  │                                                      │
│  ▼                                                      │
│  ⑤ 將中性敘述組成最終 prompt                             │
│  │                                                      │
│  ▼                                                      │
│  ┌───────────────────────────────┐                      │
│  │ 輸出：可直接送入 AI 的完整 prompt │                    │
│  └───────────────────────────────┘                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 中性敘述規則

最終給模型的文字，應偏向：

- 行為特徵描述
- 人格動能描述
- 關係互動傾向描述

避免：

- 直接出現星座語詞
- 直接暴露 code
- 讓模型知道這是一套星座分類

## 軍師型輸出規則

### 輸出定位

系統輸出應偏向：

- 分析
- 觀察方向
- 可驗證的建議

而不是偏向：

- 直接代聊
- 直接給可發送文案
- 對人下定論

### 輸出骨架

目前建議先固定骨架，不固定單一話術模板。

較合適的骨架為：

1. 觀察角度
2. 可驗證方向
3. 風險提醒

此骨架的目的不是限制最終答案長相，而是避免回答失控成以下兩種極端：

- 太空泛，導致像罐頭建議
- 太武斷，導致像替使用者下判決

### 為什麼先固定骨架而不是固定文風

因為目前仍有多項輸出問題需要測試：

- 什麼程度的具體性才會讓使用者覺得有用
- 什麼程度的抽象性才不會顯得罐頭
- 什麼程度的分析會開始接近底層 prompt 洩漏風險

因此本文件只先固定輸出骨架，不先固定最終措辭模板。

### 輸出禁止事項

目前應避免以下情況：

- 把底層 prompt 原樣或近似原樣吐給使用者
- 直接把分析做成可原封不動發送的訊息
- 直接替使用者下感情、人格或關係結論
- 把系統輸出寫成操控對方的話術

關於更細的 prompt policy 與邊界，之後在明確定 prompt 時再討論。

## 版本管理規則

星座理論未來若會調整，`theoryVersion` 必須是必填欄位。

原因：

- 同一個 `slot + code` 在不同版本下可能意義不同
- 若沒有版本欄位，舊資料會失去可解碼性
- 後續驗證分析結果時也無法知道當時用的是哪一版理論

因此開發上必須遵守：

- 新寫入的星座資料必帶 `theoryVersion`
- 對 `InternalAICopliot` 的 request 必帶 `theoryVersion`
- mapping 查找必帶 `theoryVersion`

## 錯誤處理規則

### `LinkChat` 端

若某個 slot 缺值，應明確處理，不要默默補猜。

例如：

- `sun` 缺失
- `moon` 缺失
- `rise` 缺失

應由上游流程決定：

- 是不送該 slot
- 還是拒絕此次分析

但不要在 `LinkChat` 端自行推導出替代值。

### `InternalAICopliot` 端

若 `slot + code + theoryVersion` 查不到 mapping，應回報結構化錯誤，例如：

- theory version not found
- trait mapping not found
- invalid slot
- invalid code

而不是：

- 自行改用其他 mapping
- 把未知 code 直接丟給模型猜

## 擴充規則

未來若增加更多星座欄位，應沿用同一個結構：

- 新增 slot
- 保持 code 穩定
- 照樣透過 `theoryVersion + slot + code` 查 mapping

不得因單一新需求改回：

- 直接傳中文星座詞
- 直接在 `LinkChat` 內寫 prompt
- 直接讓模型看原始分類名

本規則是正式規則，不存在「先用簡化版、之後再補正式版」的預設做法。

## 分析入口規則

分析功能的輸入形式，目前傾向走：

- 單選
- 下拉
- 多選

其目的不是單純做 UI，而是為了讓不同分析任務可以對應不同 prompt 結構。

例如不同任務可能需要不同的 prompt injection 方式：

- 團隊融入判斷
- 合作風險分析
- 相處方向分析

因此不建議一開始就把所有問題都塞進同一個萬用 prompt。

但分析入口究竟採單選、下拉還是多選，以及哪些題型先做，仍需測試，不在本文件先寫死。

## 已知困難

目前已知的產品與開發困難如下：

### 困難一：回答太安全時，容易顯得沒用

如果輸出只剩非常泛的建議，使用者會明顯覺得內容罐頭化。

### 困難二：回答太像專家斷言時，風險會升高

如果輸出直接斷定某人的性格、關係結果或合作結論，會讓系統過度武斷。

### 困難三：回答太接近底層理論時，會有洩漏風險

若系統把底層理論與 prompt 痕跡過度暴露給使用者，會損害私有理論保護。

### 困難四：不同主題可能需要不同輸出形式

不同分析主題未必適合完全相同的回答骨架與 prompt，這需要後續逐項測試。

## 待討論事項

以下項目目前保留後續討論，不在本文件先做過度承諾：

- 各分析主題的最終輸出格式
- 各分析主題的 prompt 細節
- 是否提供更多建議層次作為付費能力
- 是否增加更多 traits 系統，例如 MBTI
- 付費解鎖後 traits 如何組合進分析

## Coding Style 規範

本專案的 coding style 不追求最短、最炫或最極簡的寫法。

本專案明確採用以下取向：

- 可讀性優先
- 流程可追蹤優先
- 分層語意清楚優先
- 後續維護與 AI 協作友善優先

這代表：

- 程式可以比一般 Go 極簡派更囉嗦
- 命名可以比較長
- 註解可以比較多

只要它能換來：

- 更低的理解成本
- 更清楚的模組邊界
- 更容易交接與維護

就屬於本專案接受的風格。

### 風格總原則

#### 原則一：可讀性優先於炫技式精簡

本專案不鼓勵為了讓 code 看起來短，而把流程壓縮成難讀的寫法。

相反地，以下做法是被鼓勵的：

- 把流程拆成明確步驟
- 用中間變數保留語意
- 讓每一步的責任能被快速看懂

#### 原則二：流程要攤開，不要藏起來

若一段邏輯是主要業務流程，應盡量在 `usecase` 中明確攤開，例如：

- 驗證
- 查詢
- 建模
- 交易提交
- 跨模組同步

不應為了抽象而過早拆成大量看不出流程的 helper。

#### 原則三：囉嗦可以，但必須是有資訊價值的囉嗦

本專案接受較多註解與較長命名，但不接受以下類型的噪音：

- 單純重複程式碼表面意思的註解
- 已經過期的變更歷史長期留在正式程式中
- 無法幫助理解分層責任的裝飾性註解

也就是說：

- 好的囉嗦是降低理解成本
- 壞的囉嗦是增加視覺噪音

#### 原則四：此風格是刻意為維護與 AI 協作而設計

本專案預設會有：

- 人類開發者閱讀
- 後續自己回頭維護
- AI 協作維運

因此 code 應優先朝以下方向設計：

- 檔名可預測
- 命名可推斷
- 流程可追蹤
- 註解可補足上下文

### 命名規範

#### 命名要直白，不要過度壓縮

本專案偏好：

- `authQueryService`
- `linkUserCommandUseCase`
- `copilotQueryUseCase`

這種較長但語意清楚的命名。

不鼓勵：

- 過度簡寫
- 看不出層級與責任的短名稱

**正例 vs 反例：**

```go
// ✅ 正例：看得出模組、層級、職責
type authCommandUseCase struct { ... }
type firestoreUserRepository struct { ... }
func NewLinkCommandUseCase(...) LinkCommandUseCase { ... }

// ❌ 反例：壓縮過頭，無法快速辨識來源
type authUC struct { ... }
type userRepo struct { ... }
func NewLinkUC(...) LinkUC { ... }
```

**Interface vs Implementation 命名慣例：**

```go
// Interface：公開，PascalCase，描述能力
type AuthCommandUseCase interface { ... }
type UserRepository interface { ... }

// Implementation：私有，camelCase，可加前綴標示實作策略
type authCommandUseCase struct { ... }       // 唯一實作，不需前綴
type firestoreUserRepository struct { ... }  // 有多種可能實作，加前綴標示策略
```

**Receiver 命名慣例：**

```go
// UseCase 層用 uc 或 u
func (uc *linkCommandUseCase) ApplyLink(...) { ... }
func (u *authCommandUseCase) Register(...) { ... }

// Service 層用 s
func (s *authCommandService) HashPassword(...) { ... }

// Repository 層用 r
func (r *firestoreUserRepository) CreateUser(...) { ... }

// Handler 層用 h
func (h *AuthHandler) Register(c *gin.Context) { ... }

// Middleware 層用 m
func (m *AuthMiddleware) VerifyToken() gin.HandlerFunc { ... }
```

#### 變數名應保留層級語意

命名時應盡量保留：

- 模組
- 層級
- 用途

例如：

- `authCmdUseCase`
- `linkQueryService`
- `authMiddleware`

這能讓閱讀者快速理解依賴來自哪一層。

#### import alias 可以存在，但必須服務於分層辨識

像以下 alias 是可接受且鼓勵的：

- `cmdUseCase`
- `qryUseCase`
- `authQueryService`

前提是：

- 讓層級辨識更清楚
- 不是單純縮短字數

**正例：**

```go
import (
    cmdUC "github.com/evan0120-yo/linkchat-go/internal/auth/usecase/command"
    qryUC "github.com/evan0120-yo/linkchat-go/internal/auth/usecase/query"
    cmdService "github.com/evan0120-yo/linkchat-go/internal/auth/service/command"
    linkReq "github.com/evan0120-yo/linkchat-go/internal/link/object/req"
)
```

上述 alias 讓讀者一看就知道這是 command usecase 還是 query usecase，不需要回頭查 import path。

**反例：**

```go
import (
    c "github.com/evan0120-yo/linkchat-go/internal/auth/usecase/command"
    q "github.com/evan0120-yo/linkchat-go/internal/auth/usecase/query"
)
```

單字母 alias 省了字數但失去語意。

### 分層風格規範

本專案固定採用以下主分層：

- `provider`
- `handler`
- `usecase`
- `service`
- `repository`
- `middleware`
- `validator`
- `seeder`

#### `provider`

`provider` 是模組組裝點。

應負責：

- 建立 repository
- 建立 service
- 建立 usecase
- 建立 handler
- 回傳 module 對外依賴

不應負責：

- 業務流程
- 條件判斷
- request handling

#### `handler`

`handler` 應維持薄層。

應負責：

- bind request
- 從 context 取必要資料
- 呼叫 usecase
- 回傳 HTTP response

不應負責：

- 交易邊界
- 跨模組協調
- 核心業務規則

handler 風格上應偏向：

- early return
- 明確的 success / error response
- 不把流程塞進 handler

**正例（標準 handler 結構）：**

```go
func (h *LinkHandler) AcceptLink(c *gin.Context) {
    // Step 1: Bind request
    var body struct {
        LinkID string `json:"linkId" binding:"required"`
    }
    if err := c.ShouldBindJSON(&body); err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }

    // Step 2: 從 Context 取認證資訊
    operatorID := c.GetString(middleware.CtxKeyUserID)
    if operatorID == "" {
        c.JSON(http.StatusUnauthorized, gin.H{"error": "unauthorized"})
        return
    }

    // Step 3: 組裝 request object，呼叫 usecase
    acceptReq := &req.AcceptLinkReq{
        OperatorID: operatorID,
        LinkID:     body.LinkID,
    }
    if err := h.linkCommandUseCase.AcceptLink(c.Request.Context(), acceptReq); err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }

    // Step 4: 回傳成功
    c.JSON(http.StatusOK, gin.H{"message": "success"})
}
```

**反例（handler 內寫業務邏輯）：**

```go
// ❌ 不要在 handler 裡做驗證、查資料庫、開 transaction
func (h *LinkHandler) AcceptLink(c *gin.Context) {
    // ... bind ...
    link, _ := h.repo.FindByID(ctx, body.LinkID)  // ❌ 直接操作 repo
    if link.Status != "pending" {                   // ❌ 業務判斷放在 handler
        c.JSON(400, gin.H{"error": "wrong status"})
        return
    }
    link.Status = "active"
    h.repo.Update(ctx, link)                        // ❌ 直接寫入
    c.JSON(200, gin.H{"message": "ok"})
}
```

#### `usecase`

`usecase` 是應用流程中心，也是本專案最重要的可讀性層。

應負責：

- 明確攤開主流程
- 協調多個 service
- 協調跨模組操作
- 管理 transaction boundary

本專案鼓勵把主要流程寫成以下 5 步模式：

```text
┌─────────────────────────────────────────────────────────────┐
│                  UseCase 5 步編排流程                         │
│                                                             │
│  ① Validation（驗證）                                        │
│  │  └─ 呼叫 validator 檢查前置條件（不開 transaction）        │
│  │                                                          │
│  ▼                                                          │
│  ② Prepare（準備資料 / 查詢）                                │
│  │  └─ 呼叫 service helper（如密碼加密、生成 ID）            │
│  │                                                          │
│  ▼                                                          │
│  ③ Build（建立或修改 domain model）                           │
│  │  └─ 將準備好的資料組成完整的 domain model                  │
│  │                                                          │
│  ▼                                                          │
│  ④ Transaction（進 transaction 提交寫入）                     │
│  │  └─ RunTransaction 內執行寫入操作                         │
│  │                                                          │
│  ▼                                                          │
│  ⑤ Sync（執行同步操作）                                      │
│     └─ 在同一個 transaction 內呼叫其他模組的 usecase          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

這種形式容易閱讀與追蹤。

**正例（usecase 攤開主流程）：**

```go
func (u *authCommandUseCase) Register(ctx context.Context, cmd *req.RegisterReq) error {
    // 1. 驗證邏輯 (Validator)
    if err := u.authValidator.ValidateEmailUnique(ctx, cmd.Email); err != nil {
        return fmt.Errorf("validation failed: %w", err)
    }

    // 2. 密碼加密 (Command Service Helper)
    hashedPwd, err := u.authCommandService.HashPassword(cmd.Password)
    if err != nil {
        return err
    }

    // 3. 生成 ID (UseCase 決定 ID)
    uuidV7, err := uuid.NewV7()
    if err != nil {
        return fmt.Errorf("failed to generate uuid: %w", err)
    }

    // 4. 組裝 Domain Model
    newUser := &model.User{
        ID:          uuidV7.String(),
        Email:       cmd.Email,
        Password:    hashedPwd,
        DisplayName: cmd.DisplayName,
        Role:        model.RoleUser,
        CreatedAt:   time.Now(),
        IsActive:    true,
    }

    // 5. 交易提交 (Transaction Commit)
    err = u.client.RunTransaction(ctx, func(ctx context.Context, tx *firestore.Transaction) error {
        txAuthCommandService := u.authCommandService.WithTx(tx)
        if err := txAuthCommandService.CreateUser(ctx, newUser); err != nil {
            return err
        }
        // 跨模組同步
        if err := u.linkUserCommandUseCase.SyncUser(ctx, tx, &linkReq.CreateLinkUserReq{
            ID:          newUser.ID,
            DisplayName: newUser.DisplayName,
        }); err != nil {
            return err
        }
        return nil
    })
    if err != nil {
        return fmt.Errorf("transaction failed: %w", err)
    }
    return nil
}
```

此範例展示了 usecase 的標準結構：

1. 先驗證（不開 transaction）
2. 準備資料
3. 在 transaction 內執行寫入與跨模組同步
4. 每一步都有明確註解標示角色

**反例（把所有邏輯壓在一個函式裡不分步驟）：**

```go
// ❌ 沒有步驟標記，流程全部混在一起
func (u *authCommandUseCase) Register(ctx context.Context, cmd *req.RegisterReq) error {
    user, _ := u.authQueryService.FindByEmail(ctx, cmd.Email)
    if user != nil { return errors.New("exists") }
    pwd, _ := bcrypt.GenerateFromPassword([]byte(cmd.Password), bcrypt.DefaultCost)
    id, _ := uuid.NewV7()
    u.repo.Create(ctx, &model.User{ID: id.String(), Password: string(pwd), ...})
    u.linkRepo.Create(ctx, ...)
    return nil
}
```

#### `service`

`service` 在本專案中偏向：

- 單點領域邏輯
- repo facade
- helper logic

本專案不鼓勵把大型 orchestration 放在 `service`。

也就是：

- 薄 service 是可以接受的
- 只要它幫助責任切分，就不必勉強把邏輯塞厚

**正例（service 提供 WithTx 能力 + 單點領域邏輯）：**

```go
type AuthCommandService interface {
    WithTx(tx *firestore.Transaction) AuthCommandService
    HashPassword(password string) (string, error)
    CreateUser(ctx context.Context, user *model.User) error
    UpdateUser(ctx context.Context, user *model.User) error
}

func (s *authCommandService) WithTx(tx *firestore.Transaction) AuthCommandService {
    return &authCommandService{repo: s.repo.WithTx(tx)}
}
```

service 不自行管理 transaction，而是透過 `WithTx` 讓 usecase 決定 transaction 邊界。

#### `repository`

`repository` 應只處理 persistence。

可以負責：

- Firestore CRUD
- query
- transaction-aware persistence handling

不應負責：

- 業務驗證
- AI context 組裝
- 權限邏輯

**正例（repository transaction-aware 寫法）：**

```go
func (r *firestoreUserRepository) CreateUser(ctx context.Context, user *model.User) error {
    docRef := r.client.Collection("users").Doc(user.ID)

    // 若有 transaction，走 transaction path
    if r.tx != nil {
        return r.tx.Set(docRef, user)
    }

    // 否則走一般寫入
    _, err := docRef.Set(ctx, user)
    if err != nil {
        return fmt.Errorf("failed to create user: %w", err)
    }
    return nil
}
```

**正例（repository 查詢 Not Found 處理）：**

```go
func (r *firestoreUserRepository) FindByID(ctx context.Context, id string) (*model.User, error) {
    docSnap, err := r.client.Collection("users").Doc(id).Get(ctx)
    if err != nil {
        // NotFound 不是系統錯誤，回傳 (nil, nil) 表示「沒資料」
        if status.Code(err) == codes.NotFound {
            return nil, nil
        }
        // 其他才是真的系統錯誤
        return nil, fmt.Errorf("failed to get user: %w", err)
    }
    // ... DataTo ...
}
```

本專案在 repository 的 Not Found 處理採用 `(nil, nil)` 慣例，由上層（usecase / service）決定找不到資料時的業務語意。

#### `validator`

`validator` 用來集中前置驗證規則。

適合放：

- 唯一性檢查
- 基本合法性檢查
- 單點前置條件

不應變成：

- 跨模組 orchestration 層
- 萬用 query 入口

### Interface 與實作規範

本專案接受 interface-first 風格，尤其在以下邊界：

- usecase
- service
- repository

原因是：

- 邊界清楚
- 容易替換實作
- 有助於模組對外契約穩定

但 interface 的存在必須有語意，不應為了形式而新增無意義抽象。

**正例（interface 定義在實作檔案頂部，緊鄰實作結構）：**

```go
// Interface
type AuthCommandUseCase interface {
    Register(ctx context.Context, cmd *req.RegisterReq) error
    DeleteUser(ctx context.Context, actorID string, actorRole model.Role, targetUserID string) error
}

// Implementation
type authCommandUseCase struct {
    client             *firestore.Client
    authCommandService cmdService.AuthCommandService
    authQueryService   qryService.AuthQueryService
    authValidator      validator.AuthValidator
    linkUserCommandUseCase linkUserUseCase.LinkUserCommandUseCase
}

// Factory
func NewAuthCommandUseCase(...) AuthCommandUseCase {
    return &authCommandUseCase{...}
}
```

本專案的慣例是在同一個檔案內，由上到下排列：Interface → Implementation struct → Factory → Methods。

**反例：**

```go
// ❌ 把 interface 放在獨立檔案，遠離實作
// interfaces.go
type AuthCommandUseCase interface { ... }
type AuthQueryUseCase interface { ... }
type UserRepository interface { ... }

// ❌ 實作放在另一個檔案，閱讀時需要跳轉
// auth_usecase_impl.go
type authCommandUseCase struct { ... }
```

### Transaction 規範

```text
              Transaction 邊界圖

  ✘ 禁止開 transaction 的層                ✔ 必須開 transaction 的層
  ─────────────────────────                ─────────────────────────

  ┌──────────┐                             ┌──────────────────────────────┐
  │ handler  │  ✘ 不可在此開 transaction    │          usecase             │
  └──────────┘                             │                              │
                                           │  ✔ 決定 transaction 何時開始 │
  ┌──────────┐                             │  ✔ 在 transaction 內做       │
  │repository│  ✘ 不可偷偷做               │    跨資料與跨模組同步         │
  └──────────┘    應用層交易流程            │                              │
                                           └──────────────────────────────┘
```

這是本專案的重要固定風格。

**正例（usecase 管理 transaction 邊界）：**

```go
func (uc *linkCommandUseCase) AcceptLink(ctx context.Context, req *req.AcceptLinkReq) error {
    return uc.client.RunTransaction(ctx, func(ctx context.Context, tx *firestore.Transaction) error {
        txCmd := uc.cmdService.WithTx(tx)

        // 1. [Get] 獲取連結
        link, err := uc.queryService.GetLinkByID(ctx, req.LinkID)
        if err != nil { return err }
        if link == nil { return errors.New("link request not found") }

        // 2. [Validate] 安全性檢查
        if link.TargetID != req.OperatorID {
            return errors.New("permission denied: only the target user can accept")
        }

        // 3. [Validate] 狀態檢查
        if link.Status != "pending" {
            return errors.New("link is not in pending status")
        }

        // 4. [Modify] 修改狀態
        link.Status = "active"
        link.UpdatedAt = time.Now()

        // 5. [Update] 寫回
        return txCmd.UpdateLink(ctx, link)
    })
}
```

**反例：**

```go
// ❌ 在 handler 開 transaction
func (h *LinkHandler) AcceptLink(c *gin.Context) {
    h.firestoreClient.RunTransaction(ctx, func(...) error { ... })
}

// ❌ 在 repository 偷偷做跨集合交易
func (r *linkRepo) AcceptLink(ctx context.Context, linkID string) error {
    return r.client.RunTransaction(ctx, func(...) error {
        // 讀 link、改狀態、更新 user ... 全塞在 repo
    })
}
```

### 註解規範

#### 註解可以多，但要服務於理解

本專案接受較多註解，特別是在：

- 複雜流程
- 交易邊界
- 權限判斷
- 跨模組同步
- 效能優化原因

這類場景。

#### 註解內容應偏向三種資訊

1. 這一步在流程中的角色
2. 為什麼要這樣做
3. 若不這樣做會有什麼問題

若只是把程式本身重述一次，就不屬於高價值註解。

**正例（高價值註解）：**

```go
// [優化] 轉成 Map 方便查找 (O(M))
// 雖然有建置成本，但這能將後續查找複雜度從 O(N*M) 降為 O(N)
userMap := make(map[string]*model.LinkUser)

// 如果 err 不是 nil，代表查詢過程有錯 (DB 連線失敗等)，直接回傳錯誤
// NotFound 不是系統錯誤，回傳 (nil, nil) 表示「沒資料」
if status.Code(err) == codes.NotFound {
    return nil, nil
}
```

上述註解說明了「為什麼這樣做」和「不這樣做會怎樣」。

**反例（低價值註解）：**

```go
// ❌ 單純複述程式碼
// 建立 user map
userMap := make(map[string]*model.LinkUser)

// ❌ 重述顯而易見的邏輯
// 如果 err 不是 nil，回傳 err
if err != nil {
    return err
}
```

#### 可接受 step-by-step 註解

像以下形式在本專案是接受的：

- `1. 驗證`
- `2. 查找`
- `3. 建模`
- `4. Transaction`

因為這有助於快速追蹤流程。

#### 變更歷史註解應逐步收斂

像以下標記在快速演化期可以短期存在：

- `[新增]`
- `[修改]`
- `Phase 1`

但若功能已穩定，應逐步移出正式 code，改收斂到：

- README
- PLAN
- DEVELOPMENT
- commit history

避免正式程式長期混入過多歷史痕跡。

### 錯誤處理風格

本專案目前允許直接使用：

- `errors.New`
- `fmt.Errorf`

做清楚的錯誤表達。

偏好：

- 錯誤訊息直接可讀
- wrapping 明確
- 容易從 handler 映射成對應 response

但錯誤處理應盡量維持一致，不應同一類錯誤在不同模組有完全不同語意。

**正例（error wrapping 加上下文）：**

```go
// usecase 層：加 "transaction failed" 上下文
if err != nil {
    return fmt.Errorf("transaction failed: %w", err)
}

// service 層：加操作語意
return fmt.Errorf("hash password failed: %w", err)

// repository 層：加 persistence 語意
return fmt.Errorf("failed to create user: %w", err)
```

每一層的 error wrap 應反映該層的職責，讓錯誤鏈從上往下讀時能追蹤到出問題的位置。

**反例：**

```go
// ❌ 不 wrap，直接回傳裸 error，出問題時無法追蹤
return err

// ❌ 用 %v 而非 %w，會丟失原始 error 的型別資訊
return fmt.Errorf("failed: %v", err)

// ❌ wrap 但不加任何語意
return fmt.Errorf("%w", err)
```

**業務錯誤 vs 系統錯誤：**

```go
// 業務錯誤：使用 errors.New 或 fmt.Errorf（無 wrap）
return errors.New("permission denied: only the target user can accept")
return errors.New("link already exists or pending")
return fmt.Errorf("user not found")

// 系統錯誤：使用 fmt.Errorf + %w wrap 底層錯誤
return fmt.Errorf("failed to create user: %w", err)
return fmt.Errorf("transaction failed: %w", err)
```

### Handler Response 風格

本專案目前接受直接回：

- `gin.H{"error": err.Error()}`
- `gin.H{"data": ...}`
- `gin.H{"message": ...}`

這種簡潔回應格式。

風格上應追求：

- 一看就知道成功或失敗
- key 命名盡量一致
- handler 內 response mapping 清楚

**本專案 Response 慣例：**

```go
// 成功 - 有資料回傳
c.JSON(http.StatusOK, gin.H{"data": result})

// 成功 - 純操作確認
c.JSON(http.StatusOK, gin.H{"message": "success"})

// 成功 - 新建資源
c.JSON(http.StatusCreated, gin.H{"message": "user created successfully"})

// 失敗 - 請求格式錯誤
c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})

// 失敗 - 未認證
c.JSON(http.StatusUnauthorized, gin.H{"error": "unauthorized"})

// 失敗 - 權限不足
c.JSON(http.StatusForbidden, gin.H{"error": "forbidden: insufficient permissions"})

// 失敗 - 伺服器內部錯誤
c.JSON(http.StatusInternalServerError, gin.H{"error": err.Error()})
```

Response key 固定使用 `"data"`、`"message"`、`"error"` 三種，不混用其他 key。

### JSON Tag 規範

本專案在 JSON tag 的命名上有刻意區分：

- **Request DTO（req）**：使用 `snake_case`
- **Response DTO（resp）**：使用 `camelCase`

```go
// Request - snake_case
type RegisterReq struct {
    Email       string `json:"email" binding:"required,email"`
    Password    string `json:"password" binding:"required,min=6"`
    DisplayName string `json:"display_name" binding:"required"`
}

// Response - camelCase
type LinkItemResp struct {
    LinkID      string `json:"linkId"`
    UserID      string `json:"userId"`
    DisplayName string `json:"displayName"`
    Status      string `json:"status"`
    Direction   string `json:"direction"`
}
```

此區分是刻意設計，不是遺漏。新增 DTO 時應遵循此慣例。

#### Firestore Struct Tag 規範

Domain Model 的 firestore tag 一律使用 `snake_case`：

```go
type User struct {
    ID          string    `firestore:"id"`
    Email       string    `firestore:"email"`
    Password    string    `firestore:"password" json:"-"`  // json:"-" 排除敏感欄位
    DisplayName string    `firestore:"display_name"`
    Role        Role      `firestore:"role"`
    CreatedAt   time.Time `firestore:"created_at"`
    IsActive    bool      `firestore:"is_active"`
}
```

需要同時標示 `firestore` 和 `json` tag 的場景：

- `json:"-"`：敏感欄位（如 password）不應出現在 JSON 序列化結果中

### 常數與型別安全規範

#### 型別安全的列舉（Type-safe Enum）

本專案使用 `type X string` + `const` 定義列舉，而非裸字串：

```go
type Role string

const (
    RoleUser  Role = "user"
    RoleVIP   Role = "vip"
    RoleAdmin Role = "admin"
)

// 提供語意明確的 helper method
func (r Role) IsAdmin() bool     { return r == RoleAdmin }
func (r Role) IsVIPOrAbove() bool { return r == RoleVIP || r == RoleAdmin }
```

新增列舉類型時應遵循此模式。

#### 常數使用規則

固定值應定義為常數，避免 magic string：

```go
// ✅ 正例：Context key 用常數
const (
    CtxKeyUserID = "userID"
    CtxKeyRole   = "userRole"
)
c.Set(CtxKeyUserID, sub)

// ❌ 反例：直接寫字串
c.Set("userID", sub)
c.GetString("userID")
```

### Provider 組裝規範

`provider.go` 是模組的 DI 組裝點，負責按照正確順序建立所有元件。

**正例（完整的 provider 結構）：**

```go
func NewAuthModule(
    client *firestore.Client,
    linkUserCmdUseCase linkUseCase.LinkUserCommandUseCase,  // 外部模組依賴透過 interface 注入
) (
    *handler.AuthHandler,
    *seeder.AuthSeeder,
    *middleware.AuthMiddleware,
) {
    // 由底層往上層組裝
    // 1. Repository
    userRepo := authRepository.NewUserRepository(client)

    // 2. Service
    authQryService := authQueryService.NewAuthQueryService(userRepo)
    authCmdService := authCommandService.NewAuthCommandService(userRepo)

    // 3. Validator
    authValidator := validator.NewAuthValidator(authQryService)

    // 4. UseCase
    authQryUseCase := authQueryUseCase.NewAuthQueryUseCase(authQryService)
    authCmdUseCase := authCommandUseCase.NewAuthCommandUseCase(
        client, authCmdService, authQryService, authValidator, linkUserCmdUseCase,
    )

    // 5. Handler
    authHandler := handler.NewAuthHandler(authCmdUseCase, authQryUseCase)

    // 6. Seeder / Middleware
    authSeeder := seeder.NewAuthSeeder(client, authCmdUseCase)
    authMiddleware := middleware.NewAuthMiddleware("YOUR_SUPER_SECRET_KEY") // TODO: Config

    return authHandler, authSeeder, authMiddleware
}
```

組裝順序固定為：Repository → Service → Validator → UseCase → Handler → 周邊元件。

跨模組依賴一律透過 interface 注入，不直接引用其他模組的實作。

- 在傳輸層直接傳「太陽魔羯」這種語詞
- 在 `InternalAICopliot` 查不到 mapping 時讓模型自行猜測
- 把 slot 與 code 混成同一個欄位
- 為了追求極簡而犧牲流程可讀性
- 在主要業務流程中使用看不出責任邊界的縮寫命名
- 把 transaction boundary 分散到 handler 或 repository
- 把大型 orchestration 塞進 service
- 在正式 code 長期保留過量的歷史變更註解

## 正式推薦做法

- `slot` 使用穩定字串，例如 `sun`、`moon`、`rise`
- `code` 使用穩定代碼，例如 `a1`、`b2`、`c4`
- 所有跨專案傳輸都帶 `theoryVersion`
- 所有 prompt 最終都由 `InternalAICopliot` 組裝
- 最終送給模型的內容只保留中性人物描述

## 最終結論

在這個星座案例中，開發上應固定採用以下原則：

- `LinkChat` 保存星座位置與代碼
- `LinkChat` 只傳位置、代碼、版本
- `InternalAICopliot` 持有真正的理論解釋
- AI 不直接看到原始星座詞

這是目前最穩定、最可維護，也最符合你私有理論保護需求的做法。

## Coding Style 檢查表

新增或修改程式碼時，可用以下清單自我檢查：

### 命名

- [ ] Interface 使用 PascalCase，描述能力（如 `AuthCommandUseCase`）
- [ ] Implementation struct 使用 camelCase（如 `authCommandUseCase`）
- [ ] Factory 函式使用 `NewXxx` 前綴，回傳 interface 型別
- [ ] Receiver 變數使用短縮寫且全模組一致（h/uc/s/r/m）
- [ ] 變數名保留模組與層級語意（如 `authCmdService` 而非 `svc`）
- [ ] import alias 服務於分層辨識（如 `cmdUC`），不單純縮字

### 分層

- [ ] handler 只做 bind → 取 context → 呼叫 usecase → 回 response
- [ ] handler 不包含業務判斷、資料庫操作、transaction
- [ ] usecase 攤開主流程，步驟用數字註解標記
- [ ] usecase 管理 transaction 邊界
- [ ] service 提供 `WithTx` 能力，不自行管理 transaction
- [ ] repository 只處理 persistence，不包含業務邏輯
- [ ] provider 只做組裝，不包含條件判斷或業務流程

### 錯誤處理

- [ ] 每一層的 error wrap 反映該層職責
- [ ] 使用 `%w` wrap 系統錯誤，保留 error chain
- [ ] 業務錯誤使用 `errors.New` 或無 wrap 的 `fmt.Errorf`
- [ ] repository 的 Not Found 回傳 `(nil, nil)`，由上層決定語意
- [ ] handler 的 HTTP status code 正確對應錯誤類型

### 結構

- [ ] 檔案內順序：Interface → Struct → Factory → Methods
- [ ] import 分三段：標準庫 → 第三方 → 本專案
- [ ] Request DTO json tag 使用 `snake_case`
- [ ] Response DTO json tag 使用 `camelCase`
- [ ] Firestore struct tag 使用 `snake_case`
- [ ] 敏感欄位標記 `json:"-"`

### 註解

- [ ] 註解說明「為什麼」或「不這樣做的風險」，不複述程式碼
- [ ] 複雜流程使用 step-by-step 數字標記
- [ ] 效能優化處附上複雜度說明（如 O(N) vs O(N*M)）
- [ ] 不留長期歷史變更標記（`[新增]`、`[修改]` 等功能穩定後應移除）

### Transaction

- [ ] transaction 只在 usecase 層開啟
- [ ] transaction 內透過 `WithTx(tx)` 綁定 service / repository
- [ ] 跨模組同步在同一個 transaction 內完成
- [ ] transaction 成功回傳 `nil` 即自動 commit

### 常數與型別

- [ ] 列舉使用 `type X string` + `const` 定義
- [ ] Context key 使用常數，不使用 magic string
- [ ] 狀態比較盡量使用已定義的常數
